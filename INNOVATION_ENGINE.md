# .NeoN Innovation Engine

Innovation in `.NeoN` must be wild enough to discover new primitives and disciplined enough to become infrastructure.

This document defines how the project stays original without collapsing into feature sprawl.

## Core Innovation Thesis

`.NeoN` does not innovate by adding more features.

It innovates by discovering smaller primitives that preserve continuity across larger transformations.

```text
compression -> continuity -> proof -> topology -> inspection -> trust
```

## Innovation Rule

A new idea is not accepted because it is exciting.

A new idea is accepted when it strengthens at least one of these:

- artifact clarity
- lineage survival
- proof integrity
- local durability
- topology inspection
- symbolic compression
- prototype comprehension
- vertical coherence

## The Innovation Pipeline

```text
spark -> lens -> bridge -> experiment -> test -> demo -> vertical -> spec
```

### 1. Spark

A raw idea.

Examples:

- continuity gravity
- derivation entropy
- symbolic overlays
- ancestry-aware AI
- browser lineage runtime

A spark is allowed to be strange.

A spark is not allowed to claim production status.

### 2. Lens

A lens reframes an existing vertical without changing implementation yet.

Example:

```text
Continuity Graph viewed as evolutionary archaeology.
```

### 3. Bridge

A bridge defines how two verticals exchange meaning.

Example:

```text
Graph -> Metrics
parents + descendants -> vectors + ratios
```

### 4. Experiment

A tiny implementation or fixture.

Rules:

- small
- local-first
- inspectable
- reversible
- no dependency explosion

### 5. Test

The experiment must prove what it claims.

If it cannot be tested, it remains doctrine or future topology.

### 6. Demo

The experiment must improve the prototype loop or explain why it is deferred.

### 7. Vertical

The idea gets assigned a primary vertical owner.

### 8. Spec

Only stable behavior graduates into spec.

## Innovation Lenses

### Lens 1 — Continuity Gravity

Some artifacts become high-gravity because many derivations depend on them.

Prototype form:

```text
descendant_count + child_count + lineage_depth
```

Do not turn this into popularity.

### Lens 2 — Derivation Entropy

A lineage branch may become more complex as derivations diversify.

Prototype form:

```text
branch_count / lineage_depth
```

Future only until branching is real.

### Lens 3 — Compression Pressure

When docs, artifacts, or lineage chains become too large, they should compress into summaries or symbolic state blocks.

Prototype form:

```text
long state -> symbolic status -> human explanation
```

### Lens 4 — Proof Density

A proof packet can be measured by how much verifiable continuity it contains per file.

Prototype form:

```text
artifact hash + manifest + lineage + verification
```

### Lens 5 — Ancestry-Aware Runtime

Future runtimes should not treat data as anonymous bytes once transformed.

Future form:

```text
runtime object -> lineage metadata -> proof-aware output
```

Keep this in future topology until the CLI prototype is stable.

### Lens 6 — Symbolic Overlay

Symbols can overlay repo state, artifact state, and topology state without replacing plain language.

Prototype form:

```text
⚡ π.prototype
✓ λ.lineage
◐ λ.metrics
```

### Lens 7 — Human-Origin Preservation

AI-assisted work should preserve the human-origin chain instead of flattening authorship.

Prototype form:

```text
creator root -> workflow -> AI-assisted workflow -> proof packet
```

## Anti-Innovation Traps

Avoid:

- adding blockchain before proof primitives are stable
- adding reputation before topology is clear
- adding AI before artifact lineage is reliable
- adding UI before demo loop is understandable
- adding economics before ownership semantics are honest
- adding browser runtime before CLI prototype is solid

## Promotion Checklist

A new idea can graduate from future topology into implementation only when:

1. it has a vertical owner,
2. it has a bridge contract,
3. it can be tested,
4. it improves the demo or a verified primitive,
5. it does not weaken lower verticals,
6. it is documented honestly.
