# .NeoN Weird Layer Outline

This document captures the strange future-facing layer of .NeoN without breaking the v0.1 spine.

The primitive remains:

```text
create -> derive -> hash -> store -> export -> verify -> inspect
```

The weird layer must strengthen that loop, not distract from it.

## Rule

Weird ideas are allowed only when they improve continuity, lineage, proof, memory, attribution, or survivability.

## 1. Continuity Pressure Maps

A lineage graph can show where ideas accumulate force.

Future signals:

- derivation count
- fork density
- influence clusters
- reuse velocity
- originality pressure
- extraction pressure

Purpose:

Reveal which artifacts become gravity wells.

## 2. Semantic Lineage Distance

Not every child artifact is equally close to its parent.

Future measurements:

- surface edit distance
- structural workflow distance
- semantic divergence
- transformation depth
- synthesis score

Purpose:

Distinguish small edits from true conceptual mutation.

## 3. Continuity Fingerprints

Artifacts may develop recognizable structural signatures.

Possible fingerprints:

- workflow shape
- prompt structure
- decision path
- orchestration pattern
- concept topology

Purpose:

Track lineage even when exact text changes.

## 4. Temporal Forks

Ideas can branch into alternate timelines.

Future support:

- forks
- abandoned branches
- recombined branches
- counterfactual notes
- timeline snapshots

Purpose:

Make idea evolution inspectable.

## 5. Lineage Gravity

Some artifacts become foundational.

Possible metrics:

- descendant count
- dependency count
- attestation count
- proof strength
- survival duration

Purpose:

Identify high-gravity origin nodes.

## 6. Ancestry-Aware AI

Future AI systems should read lineage, not just text.

A model or agent could use:

- origin context
- parent artifacts
- derivation events
- proof tiers
- lineage graphs

Purpose:

Reason with continuity instead of isolated content.

## 7. Continuity-Based Reputation

Reputation should emerge from durable contribution, not popularity.

Possible signals:

- verified artifacts
- downstream derivations
- high-confidence attestations
- long-lived artifacts
- cross-system reuse

Purpose:

Build creator identity around contribution lineage.

## 8. Fossilization

Some artifacts should be frozen as reference objects.

Use cases:

- golden artifacts
- protocol primitives
- foundational workflows
- creator-defining works
- public anchors

Purpose:

Preserve civilizational memory objects.

## 9. Derivation Compression

Large lineage graphs may compress into reusable conceptual atoms.

Possible outputs:

- compressed lineage summaries
- reusable patterns
- canonical workflow atoms
- idea signatures

Purpose:

Turn complex evolution into portable primitives.

## 10. Runtime Lineage Substrate

Long-term, computation itself may carry lineage.

Future targets:

- browser capture
- Mojo-compatible runtimes
- object ancestry metadata
- lineage-aware IPC
- local verification engines

Purpose:

Move from artifact provenance to continuity-aware computation.

## Non-Negotiable Constraint

Do not implement the weird layer before the v0.1 spine is stable.

The order is:

```text
1. stable .neon artifact
2. deterministic hashing
3. CAS storage
4. proof packets
5. lineage graph
6. tests
7. golden artifacts
8. weird layer experiments
```

## Success Test

A weird feature is valid only if it answers yes to at least one question:

- Does it preserve origin?
- Does it strengthen lineage?
- Does it improve proof?
- Does it make transformation inspectable?
- Does it improve portability?
- Does it reduce attribution collapse?
- Does it help future implementations remain compatible?

If not, it belongs in the parking lot.
